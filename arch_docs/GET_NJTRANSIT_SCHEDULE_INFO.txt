# NJ Transit getTrainSchedule API Integration Analysis

## Overview
The `getTrainSchedule` API provides train schedule data for a given station, returning up to 19 upcoming trains starting from the current time. This could significantly enhance our ability to display trains scheduled later in the day that might not yet appear in the real-time DepartureVision feed.

## Key Advantages Over Current System

### 1. Extended Time Horizon
- **Current**: DepartureVision typically shows trains ~2-3 hours out
- **Schedule API**: Shows next 19 trains regardless of time span
- **Benefit**: Users can see evening trains in the morning, enabling better trip planning

### 2. Complete Stop Information
- Each train includes full stop list with times and status
- Provides pickup/dropoff restrictions at each stop
- Includes departed status for past stops
- Enables accurate journey progress calculation

### 3. Additional Data Points
- Capacity information with per-car passenger counts
- Station messages (banners, alerts, fullscreen)
- GPS coordinates and real-time position
- Stop-level delay information

### 4. Data Reconciliation Logic
```python
def merge_train_data(realtime, schedule):
    """
    Merge strategy:
    1. Use train_id as primary key
    2. Realtime data takes precedence for current status
    3. Schedule data provides future trains and stop details
    4. Combine capacity and GPS data when available
    """
    merged = {}
    
    # Add all realtime trains (authoritative for current status)
    for train in realtime:
        merged[train.train_id] = train
        
    # Add schedule trains not in realtime
    for train in schedule:
        if train.train_id not in merged:
            # This is a future train not yet in realtime
            merged[train.train_id] = train
        else:
            # Enhance existing train with schedule data
            merged[train.train_id].stops = train.stops
            merged[train.train_id].capacity = train.capacity
            
    return merged.values()
```

## API Considerations

### Rate Limiting
- Unknown rate limits on schedule API
- Implement exponential backoff
- Cache responses for 30 minutes minimum
- Stagger requests across stations

### Error Handling
```python
async def get_schedule_safe(station_code: str):
    try:
        response = await get_train_schedule(station_code)
        
        # Handle fullscreen message case
        if response.get("ITEMS") == []:
            logger.warning(f"No schedule data for {station_code}: {response.get('STATIONMSGS')}")
            return []
            
        return response["ITEMS"]
    except Exception as e:
        logger.error(f"Schedule API error for {station_code}: {e}")
        return []  # Graceful degradation
```

## Specific Implementation for Late-Day Trains

### Problem Solved
Users currently can't see evening trains in the morning because DepartureVision only shows imminent departures. The schedule API solves this by providing the next 19 trains regardless of time.

### Solution Architecture
1. **Morning Schedule Load** (5 AM):
   - Call schedule API for all major stations
   - Store all 19 trains per station
   - Mark as "schedule only" in database

2. **Progressive Enhancement**:
   - As trains approach, realtime data replaces schedule data
   - Maintain schedule data for stop lists and capacity
   - Track prediction accuracy (scheduled vs actual)

3. **User Experience**:
   - Show all trains with clear "Schedule" vs "Live" indicators
   - Display confidence levels for scheduled times
   - Allow filtering by time range

## Metrics and Monitoring

### Key Metrics to Track
1. **Coverage**: % of trains discovered via schedule vs realtime
2. **Accuracy**: Scheduled vs actual departure times
3. **Completeness**: Trains with full stop data
4. **API Health**: Success rate and response times

### Alerts
- Schedule API failures for > 1 hour
- Large discrepancies between schedule and realtime
- Missing evening trains after schedule collection

## Conclusion

Integrating the `getTrainSchedule` API would significantly enhance TrackRat's ability to provide comprehensive train information throughout the day. The 19-train limit requires strategic polling, but the additional data (stops, capacity, messages) and extended time horizon make this a valuable addition to our data collection strategy.

The key is to use it as a complementary data source that fills gaps in our current realtime-focused approach, particularly for trains scheduled later in the day that users want to plan around in advance.
